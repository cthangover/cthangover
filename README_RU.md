# cthangover — платформа для визуальных новелл с карточными боями

https://github.com/vpcoder7/cthangover/

**Жанр:** Visual Novel / Card-Battle RPG  
**Движок:** Godot 4.6 (C#, .NET 8)  
**Лицензия:** - **MIT License**
**Версия:** 0.1.0

cthangover — это не просто игра, а **мод-ориентированная платформа** для быстрого создания сюжетно-боевых визуальных новелл с элементами RPG. Главная идея: **минимальный путь от идеи до играбельной сцены**. Писатель пишет сценарий на простом DSL, он может загрузить или сгенерировать AI фон с depth-картами, система освещения адаптирует сцену под любое время суток — и сцена готова к игре.

---

## Ключевые преимущества для разработчика

### 1. Система 2D-освещения — максимум из одной картинки

Из одного фонового изображения + depth-карты система освещения извлекает максимум:

- **4 фазы времени суток** (ночь → утро → день → вечер) с плавными переходами — шейдер смешивает цветовые палитры, освещение реалистично взаимодействует с depth-картой
- **Статические источники света** — лампы, факелы, окна прописываются прямо в сценарии одной JSON-строкой: `light_set [{"x":0.3,"y":0.4,"radius":100,"color":"#ffff00"}]`
- **Лампа игрока** — draggable-лампочка, которую можно таскать мышкой по экрану, исследуя затемнённые участки. Освещение становится **игровой механикой**: спрятанные предметы, квестовые загадки, хоррор-элементы

Не нужно рисовать 4 версии каждого фона под разное время суток. Не нужно запекать свет в текстуры. Шейдер делает всё в реальном времени.

### 2. Сценарный DSL — для писателей, не для программистов

Сценарий пишется на простом языке, который поймёт геймдизайнер без опыта в коде:

```scenario
scene: dark_alley
priority: 10
condition: MainQuest.hasTag("chase_started")
---
background "city/dark_alley"
light_set [{"x":0.6,"y":0.3,"radius":80,"influence":1,"color":"#ff6600"}]

text "Фонарь у подъезда — единственный источник света." key=alley/enter
text "Ты слышишь шаги за спиной." key=alley/footsteps

select "Что делать?" key=alley/choice
  option "Обернуться" -> :turn
  option "Бежать" -> :run

:turn
action battle.init enemies=shadow_hound scene=dark_alley
switch_scene Battle
end

:run
switch_scene street_escape
end
```

**26 команд**: текст, выборы, переходы, музыка, звуки, эффекты, анимация, освещение, вызов C#-экшенов, управление квестами, добавление персонажей в группу, инициализация боёв, спавн префабов. И всё это — без единой строчки кода.

Вам этого мало? Добавляйте свои команды, просто положите cs файл с обработчиком вашей новой команды в мод, и вот, вы расширили DSL, используйте вашу новую команду в ваших сценариях, которая, к слову, теперь так же доступна и другим модам, лежащим в папке mods.

Для сложной логики — **гибридные события**: сценарий можно написать на чистом C# (через `[SceneEventAttribute]`), и он встроится в тот же пайплайн, что и `.scenario`-файлы.

### 3. Мод-архитектура с первого дня

Игра с самого начала спроектирована как платформа для модов:

- Моды — папки или `.zip`-архивы с `manifest.json`
- Моды переопределяют **всё**: персонажей, врагов, предметы, сцены, сценарии, шейдеры, текстуры, квесты, локализацию, музыку
- **Конфликт-резолюция**: позднее загруженный мод переопределяет более ранний — никакой магии
- **Include-директива**: `${food/ration}` — JSON-файлы можно собирать из кусков, переиспользуя данные между модами (путь резолвится относительно текущего файла)
- **Внешний SDK** (`Cthangover.ModSDK`): авторы модов могут писать кастомные экшены на C#, завися только от легковесной SDK-сборки, а не от всей игры. Интерфейсы `IScenarioAction`, `IActionContext`, `IQuestService`, `ICharacterService`, `IBattleService`, `ILightingService` — полный API для расширения игровой логики

Это буквально та же модель, что сделала Skyrim живым 15 лет спустя, только спроектированная с нуля в самом начале зарождения движка, а не прикрученная постфактум.

### 4. Подписки на триггеры сцен (wrapper templates + code injection)

Механизм, позволяющий моду **внедрять C#-код в сцены** при входе (`on_enter`) и выходе (`on_exit`). Сторонний мод может добавить свои UI-элементы, логику, поведение — не меняя ни строчки в основном коде.

#### Принцип работы

Шаблон (`.wrappertmpl`) — это generic C#-оболочка с плейсхолдером `{{USER_CODE}}`:

```csharp
// mods/core/wrappers/default.wrappertmpl
using Godot;
using System;

public static class SceneBuilderScript
{
    public static void Run(Node root)
    {
        {{USER_CODE}}
    }
}
```

Код подписки (`.cs`) — это чистый C#-код (тело метода `Run`), который будет вставлен в плейсхолдер:

```csharp
// mods/cooking/subscriptions/dinner_tools_show.cs
var modPanel = SceneContextNode.FindNode<Control>("ModLastPanel");
if (modPanel == null) return;

var dinner = modPanel.GetNodeOrNull<DinnerToolsBehaviour>("DinnerTools");
if (dinner == null)
{
    dinner = new DinnerToolsBehaviour();
    dinner.Name = "DinnerTools";
    dinner.Visible = true;
    dinner.SetAnchorsAndOffsetsPreset(Control.LayoutPreset.FullRect);
    modPanel.AddChild(dinner);
    modPanel.UpdateMinimumSize();
}
```

Манифест мода связывает их вместе:

```json
{
  "subscriptions": [
    { "scene": "home_kitchen", "template": "default", "code": "dinner_tools_show", "trigger": "on_enter", "priority": 20 },
    { "scene": "home_kitchen", "template": "default", "code": "dinner_tools_hide", "trigger": "on_exit", "priority": 10 }
  ]
}
```

#### Поля подписки в manifest.json

| Поле | Тип | По умолчанию | Описание |
|------|-----|-------------|----------|
| `scene` | string | — | Имя сцены, к которой привязывается подписка (ключ из `scenes/*.json`) |
| `template` | string | — | Имя шаблона-оболочки (файл `wrappers/{name}.wrappertmpl`) |
| `code` | string | — | Имя файла с кодом (`subscriptions/{name}.cs` в том же моде) |
| `trigger` | string | `"on_enter"` | `"on_enter"` — при входе на сцену, `"on_exit"` — при выходе |
| `priority` | int | `10` | Порядок выполнения (меньше = раньше) |

#### Кросс-модный поиск шаблонов

Шаблон сначала ищется в `wrappers/` того же мода, что объявил подписку. Если не найден — **ищется по всем загруженным модам**. Так, `"template": "default"` в манифесте `cooking` резолвится в `core/wrappers/default.wrappertmpl` — мод `cooking` не обязан держать свою копию шаблона.

#### Отложенное исполнение

Код подписки вызывается через `Callable.From(...).CallDeferred()` — на следующем кадре после загрузки сцены. Это гарантирует, что к моменту выполнения дерево сцены завершило layout-проход, и все родительские узлы (`ModLastPanel` и т.п.) уже имеют корректные размеры. Компиляция кода при этом синхронная и кешируется (ключ: `{ModId}_{TemplateName}_{CodeName}`).

#### Готовые шаблоны

| Шаблон | Мод | Описание |
|--------|-----|----------|
| `default` | core | Оболочка с хелперами: `Get<T>(name)`, `Get(name)`, `FindNode<T>(name)` + `Run(Node root)` с `{{USER_CODE}}` |

#### Полный пример: cooking-мод

```
mods/cooking/
├── manifest.json           # подписки на home_kitchen
├── subscriptions/
│   ├── dinner_tools_show.cs    # создаёт панель пайков при входе
│   ├── dinner_tools_hide.cs    # удаляет панель пайков при выходе
│   └── cooking_panel_init.cs   # создаёт панель готовки при входе
└── src/
    ├── DinnerToolsBehaviour.cs     # класс панели пайков
    ├── CookingPanelBehaviour.cs    # класс панели готовки
    └── ...
```

**Flow:**
1. Игрок заходит на `home_kitchen`
2. `SceneManager.SwitchScene("home_kitchen")` → `SceneSubscriptionRegistry.RunSubscriptions("home_kitchen", root)`
3. По приоритету: `cooking_panel_init` (p=10) → `dinner_tools_show` (p=20)
4. Каждый `code`-файл вставляется в `{{USER_CODE}}` шаблона `default`, компилируется через Roslyn, вызывается `SceneBuilderScript.Run(root)` отложенно на следующем кадре
5. Код создаёт `CookingPanelBehaviour` (скрытый) и `DinnerToolsBehaviour` (видимый) внутри `ModLastPanel`
6. Игрок выходит с кухни → `RunExitSubscriptions` → `dinner_tools_hide` → `dinner.QueueFree()`

### 5. Редактор сцен в Godot

Встроенный EditorPlugin (`addons/scene_manager/`) показывает граф всех сцен и сценариев **поперёк всех модов**. Визуальная валидация связей между сценами, отслеживание зависимостей и конфликтов. Не нужно держать структуру в голове или рисовать диаграммы отдельно.

## Игровые системы и механики

### 6. Система персонажей и RPG-прогрессия

Каждый персонаж (игрок и враг) обладает набором из 6 атрибутов:

| Атрибут | Назначение |
|---------|-----------|
| Health  | Здоровье |
| Attack  | Сила атаки |
| Defence | Защита |
| Strength| Сила |
| Magic   | Магия |
| Point   | Очки действий (сбрасываются каждый ход) |

- **Уровни и опыт**: персонажи получают EXP в бою, который распределяется поровну между выжившими. После боя — экран с отчётом о полученном опыте и выпавшем луте
- **Группа (party)**: до 4 персонажей в активной группе, управляется через сценарную команду `character.add_to_party`
- **Очки действий (Point)**: каждая карта действия в бою потребляет очки. Point сбрасывается до базового значения каждый ход — механика, знакомая по классическим card-battle RPG
- **AI врагов**: враги используют случайный выбор действия и цели из доступного набора, с учётом стоимости в очках и текущего состояния (оглушение блокирует ход)

### 7. Инвентарь и крафт

**Инвентарь:**
- Предметы имеют ID, имя, описание, стоимость, спрайт и **действие при использовании** (`IItemAction` — например, предмет может научить рецепту)
- **Live-обновление UI**: `PlayerInventoryBagBehaviour` подписывается на событие изменения инвентаря и автоматически перерисовывает список предметов
- Поддержка нескольких контейнеров: инвентарь игрока, контейнер вывода верстака, контейнер лута

**Рецепты и готовка:**
- Рецепт описывается как набор входных ингредиентов → выходные предметы, с указанием типа верстака и времени приготовления
- **Открытие рецептов**: использование предмета (например, волчьего мяса) может «научить» рецепту — он становится доступен в интерфейсе верстака
- **Интерфейс верстака**: показывает доступные/недоступные рецепты, подсвечивает нехватку ингредиентов красным/серым, показывает превью результата в `InventoryBagBehaviour`
- Крафт продвигает игровое время на указанное в рецепте количество минут
- Панель статистики показывает соотношение еды к количеству персонажей в реальном времени

### 8. Система отношений и рекрутинг

После победы над некоторыми врагами их можно **рекрутировать** в группу:

- **Шанс рекрутинга** (`RecruitmentChance`): задаётся в шаблоне персонажа (например, `werewolf_girl_1` — 50%)
- **Атрибуты отношений**: `Discipline` (дисциплина), `Depravity` (развращённость), `FullnessTime` (время последнего приёма пищи)
- **Голод**: параметр `Fullness` вычисляется как процент времени, прошедшего с последнего кормления, относительно окна в 24×4 часа — создаёт необходимость регулярно кормить рекрутированных персонажей
- Рекрутированные персонажи сохраняются в сейвах со всеми параметрами отношений

### 9. Система навыков

Навыки — коллекционные карты с тремя типами и пятью уровнями редкости:

| Тип | Описание |
|-----|----------|
| Active | Активный навык |
| Passive | Пассивный навык |
| ForCharacter | Привязан к персонажу |

| Редкость | |
|----------|---|
| Common | Обычный |
| Uncommon | Необычный |
| Rare | Редкий |
| Epic | Эпический |
| Legendary | Легендарный |

- Навыки загружаются из JSON модов и отображаются в виджете `SkillWidget`
- Карточки навыков имеют цветовую рамку по редкости (`SkillCardFrameBehaviour`)
- Полученные навыки сохраняются в сейвах

### 10. Сохранения и загрузка

Единый JSON-файл сохраняет **всё** состояние игры:

| Сохраняется | Детали |
|-------------|--------|
| Время | Tick игры |
| Лампа | Радиус и влияние лампы игрока |
| Персонажи | Все данные: уровень, EXP, все 6 атрибутов |
| Группа | Состав активной боевой группы |
| Инвентарь | ID предметов и количество |
| Рецепты | ID открытых рецептов |
| Квесты | Статусы и теги всех квестов |
| Slaves | Данные отношений (дисциплина, разврат, сытость) |

- Сохранения — `user://saves/{имя}.json`
- Настройки (громкость, язык) — отдельно в `user://settings.cfg` через Godot ConfigFile
- Сериализация: `System.Text.Json`, без форматирования для компактности

### 11. Система времени

Время в игре — полноценная симуляция, расширяющая 4-фазное освещение:

- **Базовая единица**: Tick = 1 игровая минута
- **Календарь**: годы (365 дней), месяцы (30 дней), дни, часы, минуты
- Продвижение: 1 игровая минута = 1 реальная секунда; также продвигается при крафте (время готовки) и программно через `AddTime()`
- Время влияет на голод (`Fullness`), освещение сцен и параметры шейдеров (`time_weights`)

### 12. Статус-эффекты в бою

Data-driven система с полным жизненным циклом:

| Тип | Эффект |
|-----|--------|
| Buff | Положительный эффект |
| Debuff | Отрицательный эффект |
| Stun | Оглушение (пропуск хода) |

- **Хуки**: `OnApply` (наложение), `OnTurnStart`/`OnTurnEnd` (с авто-декрементом длительности), `OnDealDamage`/`OnTakeDamage` (модификаторы урона через `ref int`), `OnRemove` (снятие)
- Эффекты определяются в JSON модов (`StatusEffectInfo`: тип, длительность, иконка, строка действий) и создаются фабрикой `StatusEffectActionFactory` — полностью модифицируемы

### 13. Визуальные эффекты и анимации

**Шейдерные эффекты** (помимо `timed_sprite` для освещения):

| Шейдер | Описание |
|--------|----------|
| `scene_transition` | Дизолв-переход между сценами: фрактальный шум Перлина, оранжевое свечение краёв, сдвиг оттенка, настраиваемые `progress`, `distortion`, `smoothness`, `glow_intensity` |
| `pixel_art_post_process` | Полноэкранный пост-процесс: даунсеймпл до пиксельной сетки с настраиваемым `pixel_size` |
| `pixel_art` | Попиксельный шейдер для отдельных спрайтов |
| `avatar` | Рендеринг аватаров: 5×5 kernel-обводка полупрозрачных силуэтов, конфигурируемые цвета обводки, режим `hide_mode` (сплошной чёрный силуэт) |
| `effect_anim` | Покадровая анимация спрайт-листов через `frame_index`/`frame_count` с альфа-модуляцией |

**Замена шейдеров модами:** `ShaderModReplacer` при загрузке сцены обходит всё дерево нод и заменяет `ShaderMaterial`-шейдеры и текстуры на версии из модов — любой мод может переопределить визуальный стиль без правки `.tscn`-файлов.

**Система анимаций:**
- **Покадровая** (`AnimationController`): кроссфейд между кадрами, зацикливание, настраиваемая скорость, события `NextFrame`/`NextCycle`
- **Шейдерная спрайт-листов** (`EffectAnimator`): однократное воспроизведение с автоуничтожением
- **Live2D-заглушки** (`CubismParameter`, `CubismModel`): скелет заложен для будущей поддержки Live2D-анимаций аватаров

**Диалог с двумя аватарами:** диалоговая система поддерживает одновременный показ двух аватаров (`first`/`second`) с рендерингом через `avatar.gdshader` и сменой эмоциональных выражений.

## Для кого эта платформа

- **Геймдизайнеры / писатели** — DSL позволяет собирать сцены без программирования. Написал текст, расставил свет, подключил AI-фон — сцена готова
- **Художники / AI-артисты** — предсказуемый формат аутпута (фон + depth-карта + аватары), который сразу используется движком. Depth-карта автоматически оживляет освещение. Генерация — внешний инструмент (ComfyUI), в репозиторий попадают уже готовые ресурсы
- **Программисты** — мод-архитектура, SDK, рефлексивная фабрика команд, AST-условия, гибридные события: есть где развернуться
- **Мододелы** — SDK позволяет писать расширения без зависимости от основной сборки. Мод — это просто папка/zip с manifest.json
